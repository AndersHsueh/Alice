# office mode -PM 场景方案

## 一、目标定义

`office mode -PM` 的目标，不是提供一个“适合项目经理聊天的 prompt”，而是把项目经理的高频工作流产品化为一套 **可持续复用的场景系统**。

这套系统要解决的问题包括：

- 收集团队日报
- 汇总项目进度
- 提炼阻塞与风险
- 生成 kanban 更新材料
- 生成阶段汇报和周报
- 生成或辅助维护 PMP 管理文档
- 提醒关键节点
- 追踪待办与依赖关系

## 二、为什么 PM 场景适合作为 Alice 的第一个办公垂直

### 1. 你本人就是第一用户

这意味着：

- 输入数据真实
- 需求不是猜的
- 你知道什么叫有用，什么叫无用
- 能持续迭代，而不是做一次概念展示

### 2. 输入与输出物都清晰

输入源：

- 项目组长日报
- 需求记录
- 群消息
- 会议纪要
- 进度数据
- 风险说明

输出物：

- 日报汇总
- 项目概览
- kanban 更新
- 周报 / 月报
- 风险报告
- PMP 文档草稿
- 跟进清单

### 3. 适合 agent 自动化

PM 工作中大量内容属于：

- 信息归纳
- 状态提炼
- 模板化输出
- 追踪提醒
- 跨文档整理

这恰好是 agent 最适合提升效率的地方。

## 三、产品形态建议

### 1. Mode 不是一句 prompt，而是一套工作环境

进入 `office mode -PM` 后，Alice 应该自动具备：

- 当前项目上下文
- 当前项目目录结构认知
- PM 专用模板
- PM 专用命令
- 定时任务能力
- 针对 PM 的系统提示词和输出规范

### 2. 采用“项目工作区”概念

以 Obsidian 或普通目录为基础，定义一个 PM 工作区，例如：

[`5_金碧物业项目群`](/Users/xueyuheng/tools/Obsidian/xueyuheng2026/5_金碧物业项目群)

建议在工作区内形成固定结构，例如：

- `00_项目画像/`
- `01_输入日报/`
- `02_会议纪要/`
- `03_进度与看板/`
- `04_风险与阻塞/`
- `05_汇报材料/`
- `06_PMP文档/`
- `07_自动生成/`
- `08_自动化配置/`

这样 Alice 才能稳定知道：

- 去哪里读
- 去哪里写
- 哪些文件是输入
- 哪些文件是输出

## 四、首批必须落地的能力

### 1. 日报汇总

目标：

- 扫描某一天或某一时间段的日报
- 自动按团队/模块归类
- 输出统一结构总结

推荐输出字段：

- 今日完成
- 当前进展
- 阻塞项
- 风险项
- 需要协调事项
- 明日计划

### 2. 项目风险扫描

目标：

- 从日报、任务、纪要中抽取风险与依赖
- 标记风险等级
- 给出建议动作

推荐输出字段：

- 风险标题
- 风险说明
- 来源证据
- 影响范围
- 风险等级
- 建议行动
- 责任人

### 3. 周报 / 汇报生成

目标：

- 面向不同受众生成不同口径材料

至少区分：

- 面向项目团队的周报
- 面向上级/领导的汇报
- 面向甲方/客户的简报

### 4. 待跟进清单生成

目标：

- 从日报和会议纪要中抽取下一步动作
- 输出可执行 follow-up 列表

推荐字段：

- 事项
- 来源
- 责任人
- 截止时间
- 当前状态

## 五、命令设计建议

我建议不要一开始全靠自由问答，而是给 PM 场景配明确命令。

首批建议命令：

- `/pm-daily-scan`
  - 扫描指定日期日报并生成汇总
- `/pm-risk-report`
  - 生成当前项目风险与阻塞报告
- `/pm-weekly-report`
  - 生成周报草稿
- `/pm-followups`
  - 生成待跟进事项
- `/pm-project-status`
  - 输出当前项目状态快照

这些命令本质上不是限制用户，而是保证结果稳定。

## 六、模板体系建议

项目经理场景要稳定，必须靠模板而不是完全自由生成。

建议至少维护以下模板：

- 日报归纳模板
- 风险报告模板
- 周报模板
- 领导汇报模板
- 会议纪要提炼模板
- PMP 文档模板
- 项目状态快照模板

建议把模板与项目配置分离：

- 全局模板：适合所有项目
- 项目模板：项目私有定制

## 七、自动化建议

当首批命令稳定后，再接自动化。

推荐第一批自动化：

- 每天固定时间扫描新日报并生成汇总
- 每周固定时间生成周报草稿
- 检查关键任务是否逾期
- 自动整理待跟进事项
- 通过 Feishu 推送风险提醒或日报摘要

## 八、与 Obsidian 的关系

Obsidian 不应被替代，而应被视为 PM 场景中的核心文档工作区。

Alice 的角色是：

- 读取 Obsidian 中的原始资料
- 生成结构化草稿
- 维护部分自动文档
- 帮助用户把零散信息转成可管理材料

用户仍然可以：

- 手工编辑关键文档
- 自定义目录
- 加入自己的经验和判断

所以这不是“AI 取代笔记系统”，而是“AI 成为 Obsidian 工作流里的自动化协作者”。

## 九、阶段性 MVP 建议

如果只做一个最小可用版本，我建议这样定义：

### MVP 范围

- 单个 PM 工作区
- 单项目支持
- 日报扫描
- 风险提炼
- 周报草稿输出
- 1 个定时自动任务

### MVP 不做

- 多岗位通用化
- 复杂审批流
- 组织级知识图谱
- 多项目联动总控
- 花哨 UI

## 十、最终判断

`office mode -PM` 是 Alice 当前最值得先打透的办公垂直。  
它既符合你真实的工作样本，也能形成清晰的产品边界和可验证产出。

只要你把它从“半手工工作流”推进成“目录规范 + 命令 + 模板 + 自动化”的组合，它就会成为 Alice 最强的第一批现实能力之一。

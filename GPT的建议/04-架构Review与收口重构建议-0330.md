# 架构 Review 与收口重构建议（2026-03-30）

## 结论先行

`Alice` 当前最需要的不是“大改功能”，而是一次 **架构收口**：

- 明确当前真正上线的主链路
- 把已退役但仍留在源码树里的旧实现隔离出来
- 保留 qwen-code TUI 里“当前没启用但未来可能启用”的通用能力
- 消除“同一个能力有两套实现”的混乱感

这次 review 的核心原则是：

**不是所有暂时没用到的代码都是废代码。**

尤其是：

- `src/ui/**`
- `src/commands/**`
- `src/services/**`
- `src/config/**`

这些大量来自 qwen-code 的通用 TUI/命令/UI 能力，虽然当前 Alice 业务未必全部启用，但它们仍属于 **可复用平台资产**，不应简单清理。

真正要重点处理的是：

- 已经脱离主入口的旧 REPL 链路
- 与现有主链路并行但重复的主题/命令/入口机制
- 在编译边界之外长期悬置的实验性模块

## 一、当前真实主链路

我确认的主链路是：

1. CLI 入口：[`src/index.tsx`](/Users/xueyuheng/research/Alice/src/index.tsx)

**`index.tsx -> qwen TUI -> useAliceStream -> Veronica daemon`**

所以所有 review 都应该以这条链路为准。

## 三、哪些代码不应判定为废代码

这部分很重要。

以下代码即使当前 Alice 业务还没完全用上，也不建议简单清掉：

### 1. qwen-code TUI 组件与 hooks

例如：

- `src/ui/components/**`
- `src/ui/hooks/**`
- `src/ui/contexts/**`
- `src/ui/layouts/**`
- `src/ui/themes/**`

原因：

- 这是你当前 TUI 的平台底座
- 很多暂未显式启用的能力，未来很可能直接复用
- 删除后再回补成本很高

### 2. Slash command / loader / service 体系

例如：

- `src/services/**`
- `src/ui/hooks/slashCommandProcessor.ts`
- `src/commands/**`

原因：

- 这不是单纯“没用上”的代码
- 它已经是当前 TUI 命令能力和扩展能力的通用基础设施

### 3. shim 与适配层

例如：

- `src/shim/**`
- [`src/ui/hooks/useGeminiStream.ts`](/Users/xueyuheng/research/Alice/src/ui/hooks/useGeminiStream.ts)
- [`src/shim/hooks/useAliceStream.ts`](/Users/xueyuheng/research/Alice/src/shim/hooks/useAliceStream.ts)

原因：

- 这是 Alice 把 qwen TUI 接到自己 daemon 后端的关键桥梁
- 命名上虽然还有 `Gemini` / `Qwen` 痕迹，但功能上是当前主链路的一部分

## 四、建议的收口策略

### 第一阶段：只做“标记与隔离”，不做大删改

目标：

- 不破坏现有运行主链
- 让仓库结构先清晰

建议动作：

1. 建立目录分类规则：
   - `src/runtime/` 或保持现状作为主链
   - `src/legacy-cli/` 放旧 readline/chalk REPL
   - `src/experimental/` 放当前不参与正式构建的能力
2. 给 `legacy-cli` 和 `experimental` 都加 README
3. 在主 README 或开发文档里写明当前正式主链路

### 第二阶段：消除“双实现同职责”

优先处理：

1. 主题系统
2. 版本号来源
3. 旧命令系统 vs 现役 slash command 体系

目标：

- 同一个概念只保留一个权威入口

### 第三阶段：再考虑物理删除

只有在满足以下条件时，才建议真的删代码：

- 已经迁移到 archive/legacy
- 有文档记录历史用途
- 确定 1 到 2 个版本内不会回用
- 没有测试和构建再依赖它

### 隔离

理由：

- 不该现在删除
- 但必须重新命名或标注状态，避免误导

## 六、最终判断

这次 review 的核心结论是：

`Alice` 当前最大的“废代码问题”，不是 qwen TUI 组件太多，而是 **旧自研 REPL 链路仍在主源码树里与现役主链并存**。

换句话说：

- qwen 搬来的大量 TUI 组件，大多数是 **平台资产**
- 真正需要收口的是 **退役入口、退役命令链、退役主题链**

所以这次 refactor 最好的做法不是“大删特删”，而是：

**先把历史链路隔离出来，再把主链路的边界和权威实现统一。**

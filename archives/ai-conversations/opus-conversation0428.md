> **本文件用途**：今晚（2026-04-28）我（项目作者）和 Claude Opus 4.7 进行了一次较深入的设计讨论，确立了 Alice 项目的几个核心哲学判断。本文件用于交接给 Sonnet，让你快速进入状态，不必从头解释。

## 一、Alice 是什么（一句话定位）

> **本地优先、岗位无关、Skills 可叠加、人机可接力、有环境感知和社会记忆的工作伙伴系统。**

这句话不放进 README（太像 PPT），但它是我们内部的判断锚点。

更面向用户的说法是：

> **"我不代替岗位，我让每个岗位都有自己的 Claude Code。"**

这句话**要**放进 README。

---

## 二、为什么不做"程序员版 Claude Code 的复刻"

行业现状（2026 年 4 月）：

- claude-code、qwen-code、cursor-agent、gemini、copilot CLI——**全部是给程序员的**
- 这是行业的**集体路径依赖**：程序员能容忍命令行、看得懂错误、不骂街，所以做给程序员最容易
- 但真正能让 Agent 改变社会的场景**不是程序员写代码**——程序员本就是高效率人群，提速 30% 已经很好；而让一个项目经理从每天 4 小时的文档/邮件/Kanban 里解放出来，是 300% 的效率提升

**Alice 瞄准的是企业中所有非程序员岗位**：项目经理、行政、HR、会计、产品、商务等。

---

## 三、对 Anthropic Cowork 的批评（也是我们的反向定位）

Cowork 的失败心智模型：**"什么都能 = 啥也不能"**。

普通用户打开 Cowork 看到空白对话框，会愣住——
- 我能问什么？
- 它知道我是谁吗？
- 它知道我们公司用什么准则吗？
- 我从哪个具体任务开始？

Alice 的反向设计：**用户打开就看到具体的、和他岗位相关的、当下就该做的事**。比如会计岗启动时显示"今日待处理凭证 12 张 / 本月对账还剩 3 天 / 上次报税模板需更新"。

**用户的认知负担越小，启动门槛越低，Agent 就越成功。**

## 四、核心哲学四句话（PHILOSOPHY.md 内容）

请把以下四句话作为 `docs/PHILOSOPHY.md`（或 `设计哲学.md`）的核心条款：

### 1. 我不代替岗位，我让每个岗位都有自己的 Claude Code
不站在岗位的对立面。不替代会计、HR、PM——而是让他们各自拥有一个懂自己工作流的工具。

### 2. Agent 能接住用户的工作（接力棒，不是任务交付）
绝大多数 Agent 的范式是"我来替你做完，做不了就报错"。Alice 的范式是"我做不了的，你做，但你做完了我能从那个状态继续"。

**工程后果**：Alice 的每一步必须把当前状态写进**可读文件**（Markdown / JSON / 文件名约定），不能只在内存里。用户随时可以接管、修改、推回去。这和 Claude Code 的"一个会话从头干到尾"是完全不同的范式。

### 3. 岗位是 Skills，不是写死的 Role
Opus 一开始提了一个四元组抽象（workspace_schema / toolset / workflows / greeting_strategy），被作者一句"中国私企非常草台班子"击穿了。

**真实的中国中小企业**：项目经理周一谈客户、周二写代码、周三查数据库、周四陪老板吃饭。岗位不是边界，**岗位是临时披在身上的一层皮**。

所以：
- 岗位**不能写死在 Agent 里**
- 必须用 **Skills**（可叠加、可热替换、可降级）
- 一个用户可以同时披"PM 皮 + SRE 皮 + 谈判皮"
- Skills 由用户/社区贡献，不由 Alice 内核决定

**红线**：只要 Alice 说出一句"这不是你这个岗位该做的事"，它就死了。

### 4. 云端大脑 + 本地联想（双模型架构）
这是 Alice 的**真正护城河**。

**云端模型**（Opus / Sonnet / GLM / MiniMax 经 API）：处理关键决策、复杂推理、代码生成。

**本地模型**（LMStudio + GLM-4.7-flash 32B-A3B MOE 之类）：处理"联想"——
- 持续后台扫描工作环境
- 维护人物关系图谱、事件时间线
- 在用户写邮件/文档时，主动补充被忽略的上下文

**为什么必须双架构**：
- 断网场景（飞机、地下室、客户现场信号差）
- 隐私场景（人事档案、薪资、客户名单不能上云）
- 成本场景（联想这种高频低质任务，本地模型能省 80% token）

**这三个场景恰好都是企业 IT/行政/财务最关心的**。程序员可以接受全云端，企业用户接受不了。

## 五、关于"联想能力"——一个待解决的核心问题

作者的洞察：**所有大模型（Opus 自己也承认）都缺少"联想"能力**。模型有注意力局限，被 prompt 框住，不会自发越界。但 **Agent 可以在模型外部补这个能力**。

"联想"必须拆成两类，否则会设计走偏：

### 联想类型 A：环境感知（Ambient Awareness）
对会话**外部环境**的持续监控。不需要"理解"，只需要"留意"。

工程上完全可解：后台守护进程盯着——
- token 消耗速率 vs 剩余额度（"你今晚的 token 快烧完了"）
- 网络延迟突变（节点要挂了的前兆）
- 文件系统变化（用户在外部改了文件）
- 时间线（"已经聊了 2 小时，要不要存档？"）

然后在合适时机**插入提示到对话流**。

**这件事 Alice 完全能做到，且现有所有 CLI Agent 都没做好**。优先级高。

### 联想类型 B：社会记忆（Social Memory）
"张主任刚升职，抄送也要到他，才符合礼貌。"

需要的能力：
- 持久化的**人物关系图谱**
- 时间维度的**事件记录**（"张主任 2026-03-15 升职为副总监"）
- 写邮件/文档时**主动检索**这个图谱
- **中国式礼貌逻辑**（升职要抄送、降职不能提、离职的人不能 CC、领导的领导要不要抄看场合）

这比 A 难一个数量级。但**正是 Alice 应该做的**——因为：
- 这些数据**用户私有**，不能上云 → 必须本地模型
- 这些礼貌逻辑**中国本土**，国外公司永远不会专门优化
- 这些联想需要**长期记忆** → 正好对应本地模型 + 知识库

**作者备注**：联想可能与"感受"有关，但这是哲学边缘问题，先不深入。先把 A 做出来。

## 八、待办优先级（建议）

按 Opus 视角的优先级排序，**仅供参考，最终由作者决定**：

1. **写出 `docs/PHILOSOPHY.md`**（用上面第四节的四句话）——这是后续所有 PR 的判断标准
2. **设计 Skills 系统的数据结构**：怎么发现、怎么叠加、怎么降级、怎么允许社区贡献
3. **设计"接力棒"状态持久化**：每一步的状态如何写进可读文件（Markdown / JSON）
4. **实现联想 A（环境感知）**：后台守护进程 + token/时间/文件变化监控 + 插入提示流
5. **设计本地模型集成接口**：LMStudio 兼容层，为联想 B 铺路
6. **实现联想 B（社会记忆）**：人物图谱 + 事件时间线 + 主动检索

第 1、2、3 项是**必须先于代码的**。第 4 项可以并行启动。

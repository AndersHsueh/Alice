# Alice 产品方向与路线判断

## 一句话判断

`Alice` 最有前途的方向，不是做成“另一个更换皮的 coding CLI”，而是做成一个 **在中国大陆可落地、以 daemon 为中枢、支持多前端和多工作场景的 agent harness**。

## 一、我对 Alice 价值主张的理解

### 1. 它要解决的不是单一代码问题

今天最强的一批 agent 工具，大多围绕程序员和代码仓库展开：

- `Claude Code`
- `Codex`
- 各类 IDE 内置 agent

它们很强，但也有几个现实限制：

- 中国大陆环境可用性差
- 某些产品在支付、登录、网络上直接受限
- 大量产品默认用户是程序员
- 多数工具围绕 coding，而不是完整办公流

`Alice` 的机会点恰恰在这里：

- 先解决“可用”
- 再解决“多模型可替换”
- 再解决“非程序员岗位也能直接获得 agent 帮助”

### 2. 你的核心不是 UI，而是 harness

我认为 `Alice` 的真正核心竞争力应当是：

- daemon 常驻运行
- 会话与任务编排
- 多模型接入与切换
- 工具系统
- 通道接入
- 工作模式切换
- 用户/岗位长期记忆

也就是说，`Alice` 不应该被定义为“一个终端界面”，而应该被定义为：

**一个可由 CLI、TUI、Web、Feishu 等多个前端调用的 agent runtime。**

### 3. 当前采用成熟 TUI 是对的

你已经亲自踩过终端 UI 的难点：

- 自动换行导致布局破碎
- 窗口 resize 时重绘闪烁
- 输入与流式输出互相打架
- 终端手感不稳定

这些问题很真实，也很烧研发时间。

所以当前阶段复用成熟 TUI，并把主要资源放在 harness、场景和自动化上，是正确的工程取舍，不是路线偏移。

## 二、Alice 应该怎么定义自己

我建议团队内部统一把 `Alice` 定义成三层结构：

### 1. Runtime 层

负责：

- 会话管理
- 模型路由
- 工具调用
- 任务编排
- 记忆读写
- 自动化执行
- 通道接入

这层应该尽量不依赖具体 UI。

### 2. Experience 层

面向不同交互入口：

- CLI / TUI
- Web
- IM / Feishu
- 定时任务 / 后台运行

这层只负责“怎么和用户或外部世界接触”，不负责核心推理逻辑。

### 3. Scenario 层

面向不同工作场景：

- `code mode`
- `office mode -PM`
- 未来可能的测试、运营、行政等模式

这一层才是“产品真正能卖给谁、替谁节省时间”的地方。

## 三、为什么第一阶段不要贪大

### 1. 最大风险不是技术，而是产品边界过宽

如果同时想做：

- 程序员助手
- 项目经理助手
- 通用办公助手
- 全岗位经验沉淀平台
- 多端统一产品

项目会很容易摊平在多个半成品上。

### 2. 最稳的策略是“一个垂直场景打透”

就当前已验证情况看，最适合先打透的是：

**`office mode -PM`**

原因很现实：

- 你自己就在真实使用
- 你有长期样本和持续输入
- 输出物明确
- 能快速判断产出有没有价值

### 3. code mode 不应消失，但不该独占资源

`code mode` 依然重要，因为：

- 它能帮助项目开发 Alice 自己
- 它是用户对 agent 能力最直接的期待之一

但中期来看，`Alice` 的差异化不会来自“比 Claude Code 更会写代码”，而会来自：

- 国内可用
- 多模型灵活
- daemon + 多通道
- 办公场景产品化

## 四、我建议的产品路线

### 阶段 1：把 PM 场景产品化

交付目标：

- 把你现在半手工的 PM 工作流沉淀为 `office mode -PM`
- 打通 Obsidian 工作区
- 固化模板
- 做 2 到 3 个高频命令
- 做 1 到 2 个自动化任务

### 阶段 2：把 runtime 彻底做成平台

交付目标：

- CLI / Web / Feishu 共用一套 runtime
- 会话、任务、记忆、自动化统一
- 模型切换彻底抽象成 provider/runtime 插件

### 阶段 3：复制到第二个岗位

只有 PM 场景做出稳定效果后，再复制到第二个办公岗位。  
否则太早泛化，最后会变成“所有岗位都能聊一点，但没有一个岗位真正能用”。

## 五、最终判断

我的评价是：

`Alice` 这个项目最值得坚持的，不是“自己造完整 UI 栈”，而是把 agent 从“程序员专属高级玩具”推进成“普通工作岗位也能直接使用的工作操作层”。

这条路线是成立的，也比继续卷一个纯 coding CLI 更有独立价值。

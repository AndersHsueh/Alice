# 技术架构与开发优先级建议

## 一、核心结论

接下来一段时间，`Alice` 的技术优先级应该围绕一句话展开：

**先做强 runtime 和场景，不要再把主时间消耗在自研 TUI 细节上。**

## 二、技术架构建议

### 1. 保持前后端分离

当前路线是对的：

- `Alice` CLI/TUI 是交互入口
- `Veronica` daemon 是后端中枢

建议继续强化这个边界。

daemon 应负责：

- 会话
- 模型路由
- 工具执行
- 任务编排
- 记忆
- 通道
- 自动化

前端应负责：

- 输入输出体验
- 状态展示
- 命令交互
- 审批和确认

### 2. 把 runtime 当平台来做

如果未来还要接：

- Web
- Feishu
- 定时任务
- 其他岗位场景

那么 runtime 需要有清晰模块：

- `session`
- `tools`
- `memory`
- `automation`
- `channels`
- `providers`
- `scenarios`

建议未来逐步把“模式”和“工具”都挂到这个 runtime 层，而不是写死在前端。

### 3. 模型提供方必须抽象彻底

国内环境下，模型层永远不稳定。

所以 provider abstraction 必须是第一公民，至少要支持：

- 本地模型
- 国内云模型
- 国际云模型
- 按场景选模型
- 按任务类型选模型
- fallback / degrade

对 `Alice` 来说，这比做漂亮动画重要得多。

## 三、当前阶段不建议继续重投入的方向

### 1. 自研 TUI 框架

理由不是“不酷”，而是性价比太低。

你已经验证过 TUI 底层的时间黑洞包括：

- 换行与重排
- terminal resize
- 闪烁和 repaint
- 输入体验
- 流式渲染稳定性

只要现有 TUI 壳没有明显卡住核心产品能力，就不应该现在重开一条自研 UI 战线。

### 2. 泛化到所有办公岗位

如果太早泛化，最终每个岗位都只做了 30%。  
这会让产品显得“什么都能聊一点，但什么都不能真正落地”。

### 3. 过早重做视觉层

欢迎页、logo、品牌感已经到了“足够像产品”的阶段。  
后续应当以一致性收口为主，而不是继续大幅翻新。

## 四、我建议的近期开发优先级

### P0：把 PM 场景做成第一个真正能落地的 office mode

交付物：

- PM 工作区规范
- PM 模板体系
- PM 命令集合
- 日报/风险/周报三条能力链

### P1：把 daemon 能力做成统一 runtime

交付物：

- 会话统一
- 任务统一
- 自动化统一
- 通道统一
- provider 统一

### P2：把 Web 与 IM 入口接上

理由：

- 不是所有用户都习惯 TUI
- daemon 在，前端就可以自由变化

### P3：增强长期记忆，但要结构化

这部分值得做，但一定要避免“全部对话都塞进去”的记忆垃圾化。

建议长期记忆至少区分：

- 用户画像
- 项目画像
- 岗位经验
- 工作偏好
- 稳定模板
- 历史问题样本

## 五、建议的模块演进方向

### 1. Scenario Pack 概念

我建议未来把 `office mode -PM` 设计成一个 `scenario pack`：

- 一组系统提示
- 一组工具
- 一组模板
- 一组自动化
- 一组记忆字段

这样以后复制到第二个岗位时，不需要重造核心框架。

### 2. Workspace Profile 概念

每个项目工作区都应该有一个 profile，描述：

- 项目名称
- 目录结构
- 团队角色
- 输入来源
- 输出模板
- 自动化规则
- 当前阶段

这会让 Alice 真正知道“当前正在服务哪个项目”。

### 3. 任务系统应该从 coding 扩展到办公任务

目前很多 agent 任务系统天然偏代码。  
但 PM 场景里的任务应该包括：

- 待确认事项
- 风险跟踪
- 汇报准备
- 会议纪要待落实项
- 依赖协调项

任务系统一旦抽象正确，coding 和 office 都能复用。

## 六、对团队的具体建议

### 1. 内部统一语言

建议团队把几个概念说清楚：

- `Alice`：主交互入口
- `Veronica`：daemon / orchestration / channel
- `mode`：高层工作模式
- `scenario pack`：岗位/项目场景包
- `workspace profile`：项目工作区画像

统一命名后，文档、代码和产品讨论会顺很多。

### 2. 把“半手工最佳实践”先产品化

你现在最值钱的不是抽象能力，而是已经跑通的真实 PM 工作流。  
优先把它产品化，比继续讨论更重要。

### 3. 保持工程克制

你的路线会天然诱惑团队去做很多宏大东西：

- 全岗位
- 全端统一
- 自研 UI
- 长期记忆系统
- 多 agent team

这些都能做，但不能同时做。  
当前最优策略是：**先做一个真实工作流闭环。**

## 七、最终判断

如果保持当前思路，`Alice` 完全有机会成为一个很有辨识度的产品：

- 不是只服务程序员
- 不被单一模型平台绑死
- 不被单一 UI 形式绑死
- 有 daemon 作为真正中枢
- 能往办公场景深处走

这条路线成立。  
而你现在最需要的，不是继续打磨外壳，而是把已经验证过的 PM 真实工作流沉淀成第一个成熟场景。

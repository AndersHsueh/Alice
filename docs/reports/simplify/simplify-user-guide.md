# `/simplify` 命令使用指南

## 概述

`/simplify` 命令是一个智能代码审查工具，可以自动分析你的代码变更并提供改进建议。它采用三阶段流程：

1. **阶段 1：变更检测** — 扫描 git 暂存区或最新提交的变更
2. **阶段 2：并发审查** — 三个独立的智能体并发分析代码（代码复用、质量、效率）
3. **阶段 3：聚合修复** — 去重问题、分类修复、应用自动修复

## 基础用法

### 运行完整审查

```bash
/simplify
```

默认情况下会：
- 检测你暂存区的变更
- 启动三个审查智能体分析代码
- 应用所有 minor 级别的自动修复
- 显示 major/critical 级别问题的建议修复

### 仅预览而不修改

```bash
/simplify --no-apply
```

禁用自动修复的应用。你会看到所有建议修复的预览，但文件不会被修改。

## 命令选项

### `--focus`

指定特定的审查焦点（补充默认检查）。

```bash
/simplify --focus "性能优化"
/simplify --focus "安全问题"
/simplify --focus "类型安全"
```

### `--fix-level`

控制哪些严重级别的问题自动修复。

```bash
# 仅修复 minor 级别问题（默认）
/simplify --fix-level=minor

# 修复 minor 和 major 问题
/simplify --fix-level=major

# 修复所有问题（包括 critical）
/simplify --fix-level=critical
```

### `--no-apply`

禁用自动修复的应用。只显示建议，不修改文件。

```bash
/simplify --no-apply
```

### `--no-preview`

禁用建议修复的预览显示。

```bash
/simplify --no-preview
```

### `--timeout`

设置每个审查智能体的超时时间（毫秒，默认 60 秒）。

```bash
/simplify --timeout=120000  # 120 秒
```

## 工作流示例

### 场景 1：提交前审查

```bash
# 1. 修改代码
vim src/myfile.ts

# 2. 暂存变更
git add src/myfile.ts

# 3. 运行审查
/simplify

# 4. 查看建议，决定是否接受修复
# （自动修复已应用）

# 5. 查看最终变更
git diff HEAD

# 6. 提交
git commit -m "Refactor: improve code quality"
```

### 场景 2：仅查看建议，手动修复

```bash
# 只显示建议，不修改文件
/simplify --no-apply

# 手动审查建议修复并决定是否应用
# 手动修改相关代码或忽略建议
```

### 场景 3：严格审查（应用所有修复）

```bash
# 应用 minor + major + critical 所有修复
/simplify --fix-level=critical
```

## 理解审查结果

### 问题类型

| 类型 | 描述 | 来源智能体 |
|------|------|-----------|
| **reuse** | 代码复用问题（重复代码、DRY 违规） | 复用审查器 |
| **quality** | 代码质量问题（可读性、维护性） | 质量审查器 |
| **efficiency** | 性能效率问题（算法复杂度、资源使用） | 效率审查器 |

### 严重级别

| 级别 | 说明 | 自动修复 |
|------|------|---------|
| **critical** | 可能导致错误或严重问题 | ❌ 默认不修复（需 `--fix-level=critical`） |
| **major** | 重要的改进建议 | ❌ 默认建议但不自动修复 |
| **minor** | 小的改进、最佳实践 | ✅ 默认自动修复 |

## 修复被拒绝时怎么办

如果自动修复不符合预期，可以：

1. **撤销修复** — 使用 `git checkout` 恢复原始文件
   ```bash
   git checkout src/affected_file.ts
   ```

2. **运行不修改的审查** — 下次使用 `--no-apply` 仅查看建议
   ```bash
   /simplify --no-apply
   ```

3. **手动修改** — 审查建议后手动进行必要的修改

4. **报告问题** — 如果认为修复不正确，可以反馈

## 常见问题

### Q: 为什么有些修复被应用了，有些没有？

**A:** 默认行为是只自动应用 minor 级别的修复。Major 和 critical 问题会显示为建议，需要你手动决定。使用 `--fix-level=major` 或 `--fix-level=critical` 改变这个行为。

### Q: 我不想要自动修复，只想看建议？

**A:** 使用 `--no-apply` 选项：
```bash
/simplify --no-apply
```

### Q: 三个审查器是否并发运行？

**A:** 是的！三个审查器并发运行，大幅加快审查速度。平均耗时约为单个 LLM 调用时间（通常 10-30 秒），而不是串行的 3 倍时间。

### Q: 修复失败了怎么办？

**A:** 如果修复因为代码不匹配而失败，审查器会报告错误。可以：
1. 手动进行建议的修改
2. 使用 `--no-apply` 仅查看预览，然后手动操作

### Q: 如何跳过某些文件的审查？

**A:** 当前版本会审查所有暂存变更。如果需要跳过某些文件，在运行 `/simplify` 前从暂存区移除：
```bash
git reset src/file_to_skip.ts
/simplify
```

### Q: 审查超时了怎么办？

**A:** 默认超时是 60 秒。如果某个审查器超时，其他审查器的结果仍会被应用。可以增加超时时间：
```bash
/simplify --timeout=120000  # 120 秒
```

## 最佳实践

### ✅ 推荐做法

1. **经常使用** — 在小的变更上频繁运行，而不是等待大的 PR
2. **从建议开始** — 使用 `--no-apply` 先查看建议，了解审查器的风格
3. **逐步应用** — 从 minor 修复开始，理解修复的意图后再应用更高级别
4. **提交清晰消息** — 记录哪些修改来自 `/simplify` 命令

### ❌ 避免做法

1. **盲目应用** — 不了解修复内容就自动应用
2. **混合修改** — 先运行 `/simplify`，然后手动修改后再提交（应该分开提交）
3. **忽略建议** — major/critical 问题通常代表重要改进

## 技术细节

### 修复应用的安全性

- 修复使用**精确字符串匹配**（不是正则表达式），避免意外修改
- 仅 minor 级别问题默认自动修复，降低风险
- 所有修改前会记录原始内容，支持快速回滚
- 修复失败时会报告具体原因，不会部分应用

### 去重规则

- 同一位置同一类型的多个问题会合并
- 保留最高严重级别的问题
- 例：位置 `file.ts:10` 有一个 minor 问题和一个 major 问题，最终只保留 major

### 并发架构

- 三个审查器独立并发运行（Promise.all）
- 一个审查器超时或失败不影响其他审查器
- 性能改进约 60%（相比串行执行）

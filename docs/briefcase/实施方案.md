# 公文包（Briefcase）实施方案

本文档是「项目管理公文包」功能的详细实施方案，按阶段拆分可落地项与后续依赖，并融入 Anthropic 的设计哲学与宪法式原则。

---

## 一、设计理念与哲学（Anthropic 对齐）

### 1.1 核心原则

以下原则贯穿各阶段实现与 `project-management.md` 的宪法条款，确保行为可预期、可审计、可纠偏。

| 原则 | 含义 | 在公文包中的体现 |
|------|------|------------------|
| **可审计性** | 行为依据来自可版本管理的文件，而非模型内部隐式偏好 | 初始化与行为完全由 `project-management.md`、`profile`、`pm-soul.md` 驱动；这些文件均可纳入 Git |
| **用户主权** | 凡会改变系统长期行为的决策，优先交由用户确认 | `pm-soul.md` 的写入先「建议 → 用户确认」再落盘；敏感或不可逆操作前使用 askUser |
| **渐进式复杂度** | 不强迫一次接受全部能力，按需启用、步步可见 | 阶段 1 支持「最小初始化」与「完整初始化」两种路径；每一步向用户说明「做了什么、为何需要」 |
| **结构化与可扩展** | 用 schema 与 manifest 描述能力，便于扩展新公文包类型 | `profile` 有明确 schema；公文包类型 = manifest + init 指引 + 模板，为后续 `/init product-design` 等留口 |
| **安全边界** | 自动写入 soul 等「改变后续所有对话」的操作需克制、可关闭 | 阶段 1 仅「注入 soul 到 system prompt」+「用户显式写入 soul」；自动 hook 延后至阶段 2 并可选 |

### 1.2 宪法式条款的落点

- **写入位置**：在 `project-management.md` 中设「项目宪法」小节（见该文件），作为本公文包内所有 PM 相关行为的共同约束。
- **生效方式**：在加载公文包时，将「项目宪法」与 `pm-soul.md` 一并注入 system prompt，使模型在生成、建议、写 soul 时都遵守这些条款。
- **条款风格**：参考 Constitutional AI——简短、可执行、偏好明确（如「在不确定时优先询问用户」「不静默修改约束类信息」）。

## 二、阶段划分总览

| 阶段 | 主题 | 核心交付 | 依赖 |
|------|------|----------|------|
| **阶段 1** | 可落地基础 | `/init project-management`、profile、目录骨架、模板、pm-soul 注入、宪法条款 | 无（仅需现有 ALICE/VERONICA） |
| **阶段 2** | 自动化与 soul 深化 | 心跳消费 profile、soul 更新 hook（可选）、askUser 与 soul 联动 | daemon 心跳、任务队列 |
| **阶段 3** | 扩展与规模化 | 多公文包类型、PMP 全套模板库、多 workspace 与仪表盘 | 阶段 1/2 稳定、产品规划 |

## 三、阶段 1：可落地基础（当前实施范围）

### 3.1 目标

- 用户在当前 workspace 执行 `/init project-management` 后，得到可用的「项目管理公文包」：有 profile、有目录结构、有示例模板、有 pm-soul 与宪法约束注入，无需心跳即可开始使用。

### 3.2 功能清单

1. **Slash 命令与模式边界**
   - 实现 `/init project-management`（或等价入口）。
   - **仅在 office 模式下有效**。在 coder 模式下该命令无效：输出「没有这个命令」即可，不做过度提示，用户会逐渐理解「coder 模式不做项目管理」。
   - coder 模式不提供公文包/模板能力，保持原有 CLI 功能即可。
   - 在 office 模式下行为：解析并执行「项目管理初始化指引」；指引来源见 3.3。

2. **初始化指引来源**
   - 优先读取当前 workspace 内 `./docs/briefcase/project-management.md` 或 `./agents/project-management.md`（若存在），否则使用内置/默认的 `docs/briefcase/project-management.md` 内容。
   - 确保指引文件可版本管理、可被用户修改，符合可审计原则。

3. **创建 `./profile`（JSONC）**
   - 按 `docs/briefcase/PROFILE_DESIGN.md` 与 `profile.schema.json` 生成初始 profile。
   - 必填：`name`、`briefcaseType`、`version`；选填：`tags`、`keyDocuments`、`maintenanceTasks` 等（为阶段 2 预留字段）。
   - 若已存在 profile，则校验 schema，可选「跳过 / 合并 / 覆盖」策略（建议阶段 1 仅做「已存在则跳过并提示」）。

4. **创建目录骨架**
   - `./daily-report/`：可选按 `yyyy/mm` 创建当前年月子目录；放入日报模板 `daily-yyyy-mm-dd(sample).md`。
   - `./project-analyst/`：放入 `weekly-report-Wn(sample).md`、`项目进展分析.md`、`成本统计.md` 占位或模板。
   - `./PMP/`：创建子目录名（如 `01-项目启动` … `10-周报月报`），目录内可为空或含简短 README 说明用途；暂不生成「全套 PMP 正文」（留阶段 3）。

5. **创建 `./pm-soul.md`**
   - 初始化时写入：公文包类型、当前模板与关键路径、项目管理技术偏重（由用户在 init 时选择或后续手改）。
   - 阶段 1 不实现「任务后自动写 soul」；仅实现「在进入本 workspace 或执行 PM 相关任务时，将 pm-soul.md 内容注入 system prompt」。

6. **宪法条款生效**
   - 将 `project-management.md` 中的「项目宪法」小节与 `pm-soul.md` 一起注入 system prompt（或在加载公文包时合并为一段 constitution + context）。
   - 确保所有在本公文包内的 PM 相关回复与建议遵守宪法条款。

7. **初始化完成后的摘要与行动指引**
   - init 成功完成后，输出一个摘要框 + 行动指引框，格式参考如下（项目名可从 profile 读取，临时名可提示「可改」；目录与「现在可以」的文案可依实际路径微调）：
   ```
   +------------------------------------------------+
   | 项目: project1（临时名可改）                      |
   |  目录结构:   profile          元数据文件          |
   |              project-management.md  系统规则     |
   |              pm-soul.md       自动 PM Agent      |
   |              ./daily-report/  日报               |
   |              ./project-analyst/  分析            |
   +------------------------------------------------+
   |  现在可以:                                       |
   |    把日报写在 ./daily-report/…                   |
   |    在对话中进行总结、分析项目等操作               |
   +------------------------------------------------+
   ```
   - 不替代原有「简短说明」的要点，可与上述框一起输出（如 profile/soul 路径、建议填写 keyDocuments 等）。

### 3.3 技术要点

- **profile 校验**：可用 JSON Schema 或 TypeScript 类型做轻量校验；解析时支持 JSONC 注释。
- **模板文件**：日报/周报/分析等模板可放在 `docs/briefcase/templates/` 下，init 时复制到 workspace 对应目录；或由指引中的步骤描述，由模型+工具按步骤创建（需明确「模板来自仓库」以符合可审计）。
- **soul 注入**：在 session 或 request 级别的 system prompt 组装处，若检测到当前 workspace 存在 `./pm-soul.md` 且为 project-management 公文包，则追加 soul 内容与宪法条款。

### 3.4 验收标准

- 在任意新目录执行 `/init project-management` 后，目录下出现 `profile`、`pm-soul.md`、`daily-report/`、`project-analyst/`、`PMP/`，且 profile 符合 schema。
- 在同一目录启动 ALICE 并进入 office 模式后，PM 相关对话中 system prompt 包含 pm-soul 与宪法条款（可通过调试或文档约定验证）。

### 3.5 初始化行为补充（更新3）

- **已有 `.profile` → 执行修复**
  - 检测到当前 workspace 根目录已存在 `profile` 且为合法 project-management 公文包时，不执行「新建」，而是执行**修复**：按 3.5 对「修复」的定义补缺或更新模板，且不覆盖用户已改文件。
  - 初始化完成后的摘要框显示「(修复)」，并列出本次修复项（例如：已补全某目录下缺失的模板、已按新模板版本更新某示例文件）。

- **无 `.profile` → 新建**
  - 当前目录无 `profile` 或用户已手动删除 `profile` 后再次执行 `/init project-management`，则视为**新建**。
  - 新建时，若目标路径已存在文件（如目录内已有同名模板），**直接覆盖，不提示**，以简化「决心重来」场景的操作。
  - **例外（建议至少二选一）**：对 `profile`、`pm-soul.md` 的覆盖单独处理——若该文件已存在且非本次 init 刚创建的，则**先备份再覆盖**（如 `profile.bak`、`pm-soul.md.bak`），或**提示用户确认一次**后再覆盖，避免误伤用户已填写的元数据与 soul 内容。

- **「修复」的定义（实现与文档须一致）**
  - **只补缺**：缺失的目录、缺失的模板/占位文件会创建；**已存在的文件一律不覆盖**。或
  - **按模板版本更新**：若某模板在发行版中有新版本，可用新模板覆盖工作区内**未被用户显式编辑过的**同名文件；对 `profile`、`pm-soul.md` 以及用户可能改过的文档，**不自动覆盖**。
  - 在实施方案与 `project-management.md` 中明确采用「只补缺」或「按模板版本更新」之一（或分阶段：阶段 1 仅补缺，阶段 2 再引入模板版本与覆盖策略），避免「修复」语义随版本漂移。

## 四、阶段 2：自动化与 soul 深化

### 4.1 目标

- 利用 VERONICA 心跳机制消费 `profile.maintenanceTasks`（或等价字段），定期或事件驱动地执行「待维护任务」。
- 在「任务完成」或「会话结束」时，通过可选 hook 建议对 `pm-soul.md` 的更新，经用户确认后写入，避免静默改写。

### 4.2 功能清单

1. **心跳与 profile 对接**
   - daemon 心跳时读取当前 workspace 的 `profile`，解析 `maintenanceTasks`（或 `heartbeatTriggers`）。
   - 为每个任务定义执行策略（如：生成日报、更新周报、检查里程碑），并在心跳逻辑中调用相应工具或流程。

2. **soul 更新 hook（可选）**
   - 定义「任务完成」的边界（如：某次 tool 调用结束、用户明确说「记录到 soul」、或会话结束前的摘要步骤）。
   - 在边界触发时：生成「建议写入 pm-soul 的条目」（如：新约束、干系人偏好、需人工确认项），通过 askUser 呈现给用户，用户确认后再追加到 `pm-soul.md`。
   - 配置项：允许用户关闭「自动建议 soul 更新」（仅保留手动编辑 soul）。

3. **askUser 与 soul 联动**
   - 当模型判断某操作需用户确认（如：预算变更、上线窗口、干系人敏感决策），在 prompt 中强调「此类内容应经用户确认后写入 soul」，并在实际写入时走「建议 → 确认」流程。

### 4.3 依赖与前置

- VERONICA 心跳机制已实现并可配置周期/事件。
- 阶段 1 的 profile schema 已包含 `maintenanceTasks` 或等效字段，且文档化。

### 4.4 验收标准

- 在 profile 中配置若干 maintenanceTasks，触发心跳后能按预期执行（或至少被正确解析与调度）。
- soul 的更新仅在有「建议 + 用户确认」或用户手动编辑时发生，无静默覆盖。

## 五、阶段 3：扩展与规模化

### 5.1 目标

- 支持多种公文包类型（如 `/init product-design`、`/init research-paper`），每种对应独立的 manifest + init 指引 + 模板。
- PMP 目录下提供全套 PMP 流程规范模板（或从独立模板库复制），而不仅是空目录骨架。
- 为多 workspace、多公文包列表或简单仪表盘预留数据结构与入口。

### 5.2 功能清单（概要）

- **能力包（Capability Bundle）抽象**：每种公文包类型 = 一个 manifest（名称、描述、profile 的 schema 引用、所需目录）+ init 指引 markdown + 可选模板目录。
- **PMP 模板库**：在 `docs/briefcase/templates/PMP/` 或独立仓库中维护 01–10 的完整模板内容，init 时复制到 workspace 的 `./PMP/`。
- **多 workspace / 公文包选择**：若产品需要「从列表选一个公文包打开」，则 profile、pm-soul 路径与 daemon session 的绑定在阶段 1/2 设计时已考虑，本阶段仅做 UI 与路由。

### 5.3 依赖

- 阶段 1、2 稳定；产品对多公文包类型的需求明确。

## 六、与现有代码的衔接建议

- **Slash 命令与模式**：在现有 slash 解析处注册 `init project-management`；**仅在当前为 office 模式时进入处理逻辑**，coder 模式下直接返回「没有这个命令」。处理函数内读取 `project-management.md`，按步骤调用「写文件」「创建目录」等已有或新增工具。
- **公文包检测**：在 session 或 workspace 加载逻辑中，若发现当前目录存在 `./profile` 且 `briefcaseType === "project-management"`，则判定为项目管理公文包，并加载 `pm-soul.md` 与宪法条款到 system prompt。
- **配置边界**：公文包相关路径（profile、soul、模板）尽量落在 workspace 内，不依赖全局配置；仅「是否启用 soul 建议 hook」等可选行为可放在 `~/.alice/settings.jsonc` 或 profile 内。
